# 下达任务

你已经教会了智能体规则和示例，现在是时候把工作交给它了。

## "下达任务"与"简单提问"的区别

在与智能体对话时，你的消息会被自动识别为两种意图之一：

| 类型 | 你在做什么 | 智能体的响应 | 举例 |
|------|-----------|------------|------|
| **简单提问** | 询问信息或寻求建议 | 直接回答 | "合同法中关于违约金的规定是什么？" |
| **下达任务** | 委派一件需要执行的工作 | 先制定计划，再执行 | "帮我审查这份采购合同。" |

你不需要刻意区分——智能体会根据你的表达自动判断。不过，了解这个区别有助于你更高效地使用 DesireCore。

:::tip 如何让智能体知道你在下达任务？
使用动作性的表达方式：

- "帮我审查..." / "帮我写..." / "帮我整理..."
- "把这份文件..." / "检查一下..."
- "我需要你完成..."

而不是：
- "什么是..." / "你觉得..." / "解释一下..."
:::

## 如何清晰地描述一个任务

好的任务描述让智能体能更快、更准确地完成工作。一个完整的任务描述包含四个要素：

### 1. 做什么（目标）

明确说出你要的结果：

```
你："帮我审查附件中的采购合同，输出一份审查报告。"
```

### 2. 怎么做（标准）

如果你有特定的要求或标准：

```
你："审查重点关注违约条款、付款条件和质保期。
    报告格式按上次教你的那个模板来。"
```

:::info 如果你之前教过规则
如果你已经教过智能体审合同的规则，这一步可以省略——它会自动应用已学的规则。
:::

### 3. 范围和边界

让智能体知道什么该做、什么不该做：

```
你："只看合同主体部分，附录和附件不用审查。
    如果发现重大风险，先告诉我，不要直接给对方发修改建议。"
```

### 4. 时间和优先级（可选）

如果有紧迫性要求：

```
你："这份合同比较急，今天下午 3 点前需要审查结果。"
```

## 好的任务描述示例

:::tip 好的任务描述 vs 模糊的任务描述

**好的描述**：
> "帮我审查附件中的'XX科技采购合同.pdf'。重点检查违约条款和付款条件是否符合公司标准。输出格式按之前教你的审查报告模板。如果有高风险条款，在报告中标红。"

**模糊的描述**：
> "看看这个合同有没有问题。"

---

**好的描述**：
> "帮我给张总写一封邮件，告诉他下周二的会议改到周三下午 2 点，因为会议室冲突了。语气正式但友好，简短即可。"

**模糊的描述**：
> "帮我写个邮件。"

---

**好的描述**：
> "整理上周的销售数据（附件中的 Excel），按区域分组统计总销售额，并找出同比增长最快的 3 个区域。输出为一个简洁的数据表格加一段分析总结。"

**模糊的描述**：
> "帮我分析一下数据。"
:::

## 任务下达后会发生什么

当你下达一个任务后，智能体会按照以下流程运作：

```
你下达任务
    ↓
智能体可能追问（Clarify）—— 如果信息不够
    ↓
智能体制定计划（Plan）—— 并等你确认
    ↓
你确认或修改计划
    ↓
智能体开始执行（Execute）
    ↓
遇到关键节点时暂停确认（人闸门）
    ↓
执行完毕，提交回执（Reflect）
```

每一步你都能看到、能控制。这就是"委派"和"丢给 AI 就不管了"的区别。

## 任务的范围控制

你可以根据需要控制智能体的自主程度：

| 控制程度 | 适用场景 | 表达方式 |
|---------|---------|---------|
| **全自主** | 低风险、重复性任务 | "直接帮我做就行，不用每步都问我" |
| **关键节点确认** | 中等风险任务 | "重要的决策问我，其他你自己判断" |
| **每步确认** | 高风险或首次执行 | "每一步都跟我确认后再做" |

:::
