# 心跳监控

传统 AI 助手通常只会等待你的指令。DesireCore 的心跳（Heartbeat）让智能体可以定期“睁开眼睛看一看”：按你配置的频率检查邮箱、代码仓库、日历、系统事件或其他连接器，判断是否有需要你关注的变化。

心跳不是普通定时任务的一个动作类型。它由独立的心跳调度器管理，不写入 `schedules/*.json`，也不需要你创建一条“每 30 分钟执行心跳”的定时任务。你可以在智能体详情页的心跳区域单独启用、暂停、配置和查看历史。

## 什么是心跳

心跳的核心配置来自智能体仓库中的文件：

```text
~/.desirecore/agents/<agent_id>/heartbeat/HEARTBEAT.md
```

这个文件不是普通备注，而是智能体执行巡检时必读的检查清单。它应该说明：

- 要检查哪些数据源或系统事件
- 每个数据源的重要程度
- 哪些变化只需要记录 OK
- 哪些变化需要产生 Alert
- 输出结论时应包含哪些摘要、详情和建议

如果某个智能体没有 `HEARTBEAT.md`，系统启动时会自动创建一个空模板。保持文件为空或只有注释时，心跳不会发起实际 LLM 检查，也不会把这次空检查当作有效巡检累计；添加明确检查项后，心跳才有实质内容可执行。

## 执行流程

每次心跳会经历以下步骤：

1. **读取配置**：合并用户偏好、智能体默认值和 `heartbeat/HEARTBEAT.md`。
2. **检查抑制策略**：判断是否处于 Snooze、专注模式、安静时段、冷却期或忙碌状态。
3. **构建上下文**：加载心跳检查清单、偏好配置和近期系统事件。
4. **执行查询**：向智能体发送心跳提示词，要求逐项检查需要外部数据的来源。
5. **解析结果**：识别 `HEARTBEAT_OK` 或解析结构化 JSON / 文本结论。
6. **记录回执**：写入可追溯的心跳回执，并根据结果决定是否通知。

如果当前有活跃用户查询，后台心跳会跳过本次执行，避免抢占正在进行的对话资源。手动触发心跳时会跳过抑制检查，因为这代表你明确想立即查看结果。

## 两种结果

心跳采用 OK / Alert 两态模型：

| 结果 | 行为 | 说明 |
|------|------|------|
| **OK** | 静默记录 | 所有检查项都正常，或智能体明确回复 `HEARTBEAT_OK` |
| **Alert** | 插入对话流卡片 | 有变化、异常、风险、建议或需要你阅读的结论 |

Alert 覆盖“需要阅读”和“可能需要后续操作”的情况。是否需要你进一步确认，取决于 Alert 内容和后续操作本身的审批机制：例如发送邮件、执行命令、写文件等外部影响操作仍会走独立审批卡片，不会因为心跳触发而绕过确认。

### OK

OK 表示“检查过了，没有需要打扰你的事项”。它会写入心跳历史和回执，但默认不会推送通知。

在对话流中，OK 会显示为绿色可折叠 pill，内容类似“心跳检查 · 一切正常 · 检查了 N 项”。点击后可以展开查看各数据源名称、摘要和总结论。你也可以在心跳历史里看到它的时间、结论和是否被抑制。

### Alert

Alert 表示“有内容值得你看”。它可能来自：

- 数据源出现新变化
- 工具或连接器检查失败
- 安静时段后需要汇总积累变化
- 智能体发现风险、异常或建议
- 定时任务或系统事件进入心跳上下文后需要你关注

Alert 会尽量展示最终结论、来源摘要和建议，避免把中间工具输出直接塞进通知正文。重复内容会在 24 小时窗口内去重，避免同一事项反复打扰。

在对话流中，Alert 会显示为心跳通知卡片。卡片通常包含：

- 每个数据源的名称、变化数量、摘要和明细
- 智能体的总结论
- 后续建议列表
- **稍后提醒** 菜单，常用选项为 15 分钟、1 小时和 3 小时

Alert 卡片只负责提醒和说明，不直接替代审批。若 Alert 后续需要发送邮件、改文件、执行命令或调用外部服务，系统会另外显示对应工具的审批卡片。

## 配置巡检

在智能体详情页的心跳区域点击设置按钮，可以打开心跳设置面板。

![心跳设置面板](./img/heartbeat-settings.png)

### 基本设置

| 设置项 | 说明 |
|--------|------|
| **启用心跳** | 开启或关闭该智能体的心跳调度 |
| **检查间隔** | 支持 15 分钟、30 分钟、1 小时等常用间隔 |
| **编辑检查内容** | 跳转到 AgentFS 中的 `HEARTBEAT.md`，修改具体巡检任务 |

心跳偏好由用户级 `heartbeat.json` 和智能体级 `HEARTBEAT.md` 默认值合并得到。系统级最短间隔为 15 分钟；`HEARTBEAT.md` 还可以声明更高的 `min_interval` 约束，用户选择的间隔不能低于该约束。

### 安静时段

安静时段包含开始时间和结束时间，默认通常是 `23:00` 到 `08:00`。它支持跨午夜设置，并按配置时区解释。

处于安静时段时，心跳属于硬抑制：系统跳过 LLM 检查，只写入“被抑制”的回执，原因会记录为安静时段。安静时段结束后的首次心跳会进入合并汇报模式，系统会在上下文中提示智能体综合检查静音期间可能积累的变化，而不是只看最近增量。

### 通知控制

| 控制项 | 行为 |
|--------|------|
| **冷却时间** | 限制连续通知频率；心跳仍会执行，但重复通知会被抑制 |
| **专注模式** | 硬抑制；只记录，不执行 LLM 检查，不推送通知 |
| **Snooze** | 临时静音；支持静音 1 小时或静音到明天 8:00 |
| **忙碌信号** | 你正在实时对话时照常执行检查，但通知由上层延后或抑制 |

硬抑制包括 Snooze、专注模式和安静时段。命中这些策略时，心跳不会消耗模型调用，只写回执。冷却期和忙碌信号属于软抑制：检查仍执行，但通知展示会被控制。

## 查看巡检结果

心跳启用后，智能体详情页会显示状态卡片和历史卡片。当心跳未启用时，页面会显示一个简化入口，你可以从那里重新打开设置。

### 心跳状态卡片

状态卡片展示当前巡检状态：

- **运行状态**：活跃、已静音、专注模式、安静时段或其他抑制原因。
- **上次检查**：最近一次心跳执行的相对时间。
- **下次检查**：独立心跳调度器计算出的下一次触发时间。
- **今日检查**：当天已记录的心跳次数。
- **状态分布**：当天 OK 和 Alert 的数量。
- **待配置提示**：如果 `HEARTBEAT.md` 没有实质检查项，会提示你去设置中编辑。

![心跳状态卡片](./img/heartbeat-status-card.png)

### 心跳历史

历史卡片列出最近心跳回执，默认显示最近 5 条，展开后最多加载 50 条。你可以按全部、OK、Alert 筛选。

每条回执通常包含：

- 执行时间
- 结果级别
- 结论摘要
- 数据源摘要和变化数量
- 抑制状态和原因
- 失败原因（如果有）

每次心跳都会生成可追溯回执。即使命中硬抑制，系统也会记录 `suppressed: true`、抑制原因和时间，保证审计链完整。

## 数据源和检查内容

心跳并不内置固定的数据源列表。它依赖智能体拥有的工具、MCP 服务、邮箱或仓库连接器，以及 `HEARTBEAT.md` 中写明的检查规则。

一个有效的检查清单通常包含：

```md
# 心跳检查

## 数据源
- 邮箱：检查来自客户、CEO 或项目关键人的未读邮件
- GitHub：检查被 @、PR 审查请求和失败的 CI
- 日历：检查 24 小时内新增或冲突的会议

## 判断标准
- 没有新增重要事项时回复 HEARTBEAT_OK
- 有需要我阅读的信息时输出 Alert 摘要
- 需要外部操作时只提出建议，不直接执行
```

如果心跳没有按预期检查某个来源，优先确认两件事：`HEARTBEAT.md` 是否明确写了该来源，以及对应工具、MCP 服务或账户连接是否可用。

## 静音、去重和恢复

心跳系统会尽量克制地通知：

- 静音时段、Snooze 和专注模式会跳过模型检查并记录抑制回执。
- 冷却时间会避免短时间内重复通知。
- 忙碌信号会在你正在对话时降低打扰。
- 24 小时内容去重会抑制重复 Alert。
- 应用重启后，调度器会从最近心跳回执恢复上次执行时间，避免每次启动都从零重新计时。
- 如果没有任何智能体显式开启心跳，系统会为默认智能体自动开启一个 30 分钟间隔的心跳偏好。

## 瞬态错误处理

心跳执行 LLM 查询时会对瞬态错误自动重试。典型瞬态错误包括限流、网络断开、超时、服务暂不可用和 5xx 错误。

重试策略最多尝试 4 次，退避间隔依次为 3 秒、8 秒、20 秒和 60 秒。如果仍然失败，本次心跳会记录失败原因；周期性心跳不会因为单次失败而丢失配置。

## 与定时任务的区别

| 能力 | 适合做什么 | 结果呈现 |
|------|------------|----------|
| 心跳 | 持续巡检“是否有变化需要关注” | OK 静默，Alert 通知 |
| 定时任务 | 到点执行一段明确 Prompt | 执行结果进入任务记录、会话或通知 |
| Sleep | 等待几秒后继续当前任务链 | 仍在当前会话中继续 |

如果你想“每 30 分钟看看有没有重要邮件，并且没事别打扰我”，使用心跳。如果你想“每天 9 点生成一份日报”，使用[定时任务](./03-scheduled-tasks.md)。如果只是让智能体等几秒再继续当前操作，使用 `Sleep`。
